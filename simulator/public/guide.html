ATtiny85 USI — I2C Transaction Reference

Phases
  • IDLE — Bus resting state
  • INIT — USI peripheral configuration
  • START — Claiming the bus
  • ADDR — Sending the device address
  • ACK — Slave acknowledgement
  • DATA — Sending a data byte
  • STOP — Releasing the bus

IDLE

# idle-bus — Bus Idle State

When no device is communicating, both SDA and SCL float HIGH, held there by external pull-up resistors (typically 4.7kΩ to VCC). No device is actively driving either line. This is the resting state of the I2C bus and the condition that every device checks before initiating a transaction.

The pull-up resistors are essential — I2C uses open-drain signaling, meaning devices can only pull lines LOW. They release the line and let the resistor pull it back HIGH. If you forget the pull-ups, both lines will float and nothing works.

Common mistake: Forgetting external pull-up resistors. Internal pull-ups on the ATtiny85 are too weak (~50kΩ) for reliable I2C. You need 4.7kΩ external resistors on both SDA and SCL.

INIT

# init-usidr — Loading USIDR and Configuring USICR

The i2c_init() function starts by loading USIDR with 0xFF. This ensures that when the SDA pin is configured as an output, it drives HIGH — matching the idle bus state. If USIDR held 0x00, enabling the output would yank SDA LOW and generate a false START condition.

Next, USICR is configured for two-wire (I2C) mode. The key bits are USIWM1:0 = 10 (two-wire mode, which connects USIDR bit 7 to the SDA pin) and USICS1:0 = 10 (software clock strobe — the firmware toggles SCL by writing to USITC).

USICR = (1 << USIWM1) | (1 << USICS1) | (1 << USICLK);
Common mistake: Forgetting to pre-load USIDR with 0xFF before setting the pin as output. This causes a glitch on SDA that looks like a START condition to other devices.

# init-usisr — Clearing the Status Register

USISR is split into two halves. The upper nibble (bits 7:4) contains interrupt flags: USISIF (Start), USIOIF (Overflow), USIPF (Stop), and USIDC (Data Collision). The lower nibble (bits 3:0) is the 4-bit clock edge counter.

To clear the flags, you write 1 to each flag bit — this is a standard AVR "write-1-to-clear" pattern. Writing 0 to the counter bits resets the counter. The counter counts both rising and falling edges of SCL, so it takes 16 edges (8 rising + 8 falling) to transfer one byte — at which point it overflows from 15 to 0 and sets USIOIF.

USISR = (1 << USISIF) | (1 << USIOIF) | (1 << USIPF);
Common mistake: Trying to clear flags by writing 0 to them. On AVR, interrupt flags are cleared by writing 1, not 0. Writing 0 leaves them unchanged.

START

# start-sda-fall — SDA Falls While SCL Is HIGH

The I2C START condition is defined as SDA transitioning from HIGH to LOW while SCL remains HIGH. This is the universal "attention" signal on the bus — every device monitors for it. The master generates this by clearing the PORTB bit for SDA while SCL is still HIGH, which drives SDA LOW through the open-drain output.

The USI hardware automatically detects this transition and sets the USISIF (Start condition Interrupt Flag) in USISR. In polled mode (no interrupts), the firmware checks this flag to confirm the START was generated successfully.

During normal data transfer, SDA is only allowed to change while SCL is LOW. The START condition is special precisely because SDA changes while SCL is HIGH — that is what makes it distinguishable from regular data.

# start-scl-fall — SCL Pulled LOW After START

Immediately after the START condition, the master pulls SCL LOW. Now both SDA and SCL are LOW. This marks the beginning of the first clock cycle and signals to all devices that a transmission is about to begin. The bus is now claimed — no other master may transmit until a STOP condition releases it.

From this point forward, the master controls SCL by toggling it with software strobes (writing to USITC in USICR). The master will toggle SCL for each bit of the address byte, then the ACK bit, then data bytes, until it issues a STOP.

Common mistake: Not waiting long enough between pulling SDA LOW and pulling SCL LOW. The I2C spec requires a minimum hold time for the START condition (typically ~4μs at 100kHz). Without a brief delay, fast masters may violate timing requirements.

ADDR

# addr-load-usidr — Loading the Address Byte

The master loads the 8-bit address frame into USIDR. I2C uses a 7-bit device address occupying bits 7:1, with bit 0 as the R/W flag (0 = write, 1 = read). So the address is shifted left by one and OR'd with the direction bit.

USIDR = (slave_addr << 1) | 0; // write to 0x3C → 0x78

The master also sets DDRB for the SDA pin to OUTPUT so it can drive data onto the bus. In two-wire mode, bit 7 of USIDR is physically connected to the SDA line, so whatever is in the MSB appears on the wire immediately.

Common mistake: Sending the 7-bit address without shifting it left. If your device is at address 0x3C and you load 0x3C into USIDR, you are actually addressing device 0x1E with a read bit — not what you want.

# addr-clock-rising — Rising Edge: Slave Samples SDA, USIDR Shifts

When the master toggles SCL HIGH (by writing 1 to USITC in USICR), the slave device samples the SDA line. Whatever logic level SDA holds at this moment is the bit value the slave reads. This is why SDA must be stable before SCL rises — changing SDA while SCL is HIGH would be interpreted as a START or STOP condition.

With USICS1=1 (external positive-edge clock source), the USI shifts USIDR left by one bit on this rising edge. The bit that was in position 6 moves to position 7 (the MSB), and since bit 7 is connected to SDA in two-wire mode, the next data bit now appears on the wire. This automatic shift is the key advantage of the USI — you load the byte once and the hardware handles the bit-by-bit serialization.

The 4-bit counter in USISR also increments by 1 (via the USICLK software strobe written in the same USICR write).

# addr-clock-falling — Falling Edge: Counter Increments Only

When SCL falls back LOW, the counter increments again (via another USICLK strobe), but the data register does not shift — with USICS1=1, shifts only happen on positive (rising) edges. SDA remains stable showing the next bit to send.

After one complete bit (one rising + one falling edge), the counter has increased by 2. A full byte takes the counter from 0 to 16, which causes an overflow.

# addr-overflow — Counter Overflow After 8 Bits

After 8 bits have been clocked out (16 edges: 8 rising + 8 falling), the 4-bit counter overflows from 15 back to 0. This overflow sets the USIOIF (Counter Overflow Interrupt Flag) in USISR. The firmware polls this flag in a tight loop to know when the byte transfer is complete.

At this point, the address byte has been fully transmitted. The slave that recognizes its address will prepare to send an ACK on the next clock cycle. The master must now release SDA and clock one more bit to read the acknowledgement.

while (!(USISR & (1 << USIOIF))) { /* toggle SCL */ }

ACK

# ack-release-sda — Master Releases SDA

After transmitting 8 bits, the master must let go of SDA so the slave can respond. It does this by switching DDRB for the SDA pin to INPUT (0), which tri-states the pin — the master is no longer driving it. The external pull-up resistor pulls SDA HIGH by default.

If a slave recognized the address, it will actively pull SDA LOW during the next clock cycle. If no slave responds (wrong address, device missing, or device busy), SDA stays HIGH from the pull-up. This is the difference between ACK (LOW) and NACK (HIGH).

The counter is pre-loaded to 14 (0x0E) so that it overflows after exactly 2 more edges (14 → 15 → 0), which equals one clock cycle — just enough to read the single ACK/NACK bit.

USISR = (1 << USIOIF) | (1 << USIPF) | 14; // 1-bit transfer

# ack-clock-rising — Reading the ACK Bit

The master toggles SCL HIGH for the 9th clock cycle. On this rising edge, the master reads SDA via USIDR bit 0 (the input shift). If the slave is holding SDA LOW, the bit reads as 0 — this is an ACK, meaning the slave received the address and is ready for data.

If SDA reads HIGH, it is a NACK. This means either no device has that address, the device is busy, or there is a wiring problem. The master should typically issue a STOP condition and retry or report an error. In the simulator, you will see a green SDA line for ACK (slave pulling LOW) or the line staying HIGH for NACK.

Common mistake: Forgetting to pre-load the counter to 14 for the ACK bit. If the counter starts at 0, the master will clock 8 bits instead of 1, corrupting the transaction.

# ack-clock-falling — Completing the ACK Cycle

SCL falls LOW, completing the ACK clock cycle. The counter overflows (14 → 15 → 0) and sets USIOIF in USISR. The firmware detects this overflow and reads the result from USIDR to determine ACK or NACK.

After reading the ACK, the master reclaims SDA by setting DDRB for the SDA pin back to OUTPUT. It is now ready to load the next byte into USIDR and transmit data, or to issue a STOP condition to end the transaction.

DATA

# data-load-usidr — Loading the Data Byte

The process is identical to the address phase. The master loads the data value (for example, 0x42) into USIDR and resets the counter in USISR to 0. The USI hardware does not distinguish between address bytes and data bytes — that distinction exists only at the protocol level. From the hardware's perspective, it is always just shifting out 8 bits and counting 16 edges.

Multiple data bytes can be sent in sequence, each followed by an ACK cycle. The slave address is only sent once per transaction (after the START condition). Everything after the first ACK is data until the master issues a STOP.

# data-clock-rising — Rising Edge: Slave Samples Data

Identical to the address phase. When SCL goes HIGH, the slave samples SDA and captures the current bit value. The counter increments by 1. The master must ensure SDA is stable before raising SCL — all data transitions happen while SCL is LOW.

Each rising edge is a "capture moment." If you looked at the bus with a logic analyzer, you would see SDA change only while SCL is LOW, and remain stable during the entire HIGH period of SCL. This discipline is what keeps data transfer reliable on a shared bus.

# data-clock-falling — Falling Edge: USIDR Shifts

Identical to the address phase. The USI hardware shifts USIDR left by one bit on each falling edge, placing the next bit in the MSB position (bit 7) where it connects to SDA. The counter increments again. After this edge, the next bit to transmit is already on the wire, waiting for the next rising edge when the slave will sample it.

# data-overflow — Counter Overflow After 8 Data Bits

After 16 clock edges (8 complete bit cycles), the counter overflows and sets USIOIF. The firmware detects this and knows the data byte is fully transmitted. The flow then returns to the ACK phase — the master releases SDA, pre-loads the counter to 14, and clocks one more bit to read the slave's acknowledgement.

This cycle of load-clock-overflow-ACK repeats for every data byte in the transaction. A single I2C write transaction might send one byte or hundreds — the protocol is the same each time.

STOP

# stop-sda-low — Force SDA LOW

The STOP condition requires SDA to rise while SCL is HIGH — the mirror image of a START. To guarantee a clean rising edge, the master first forces SDA LOW (it may already be LOW from the last data bit or ACK, but we ensure it). This is done by clearing the PORTB SDA bit and setting DDRB SDA to OUTPUT.

At this point, SCL is still LOW from the last clock cycle. Both lines are now LOW, which is the setup for the STOP sequence.

# stop-scl-rise — Release SCL

The master releases SCL by either toggling it via USITC or by switching the SCL pin to input and letting the pull-up resistor bring it HIGH. SCL rises while SDA remains LOW. This is not yet a STOP condition — SDA changing while SCL is HIGH is what defines START and STOP. Right now SCL has changed, not SDA.

The master must wait for SCL to actually reach a HIGH level before proceeding. In multi-master or clock-stretching scenarios, a slow slave could hold SCL LOW. For a single-master bus with no clock stretching, a brief delay is sufficient.

# stop-sda-rise — SDA Rises: STOP Condition

Finally, the master releases SDA while SCL is HIGH. SDA rises via the pull-up resistor, generating the STOP condition. Every device on the bus recognizes this transition and knows the bus is now free. The USI hardware sets USIPF (Stop condition Flag) in USISR.

The bus returns to the IDLE state — both lines HIGH, no device driving. Another transaction can begin after the required bus-free time (typically ~4.7μs at 100kHz). The I2C transaction is complete.

Common mistake: Releasing SDA before SCL is HIGH. If SDA rises while SCL is still LOW, it is just a normal data transition — not a STOP condition. The slave will not recognize it and the bus will be stuck in a busy state.
